Auto-Update

Farx checks for updates on every startup. When an update is available, you'll see a notification. To update:

farx --update

Quick Start

Run farx in your terminal. You'll see two panels side by side showing your current directory (left) and home directory (right).

  • Use Up/Down to navigate, Enter to open directories
  • Tab switches between panels
  • F3 views a file, F4 edits it
  • F5 copies, F6 moves, F8 deletes
  • F1 opens the full help screen
  • F10 quits

Search

Press Alt+F7 to open the search dialog. You can search by filename pattern (glob) and/or by content. Results are listed in a scrollable list — press Enter to jump to the directory containing the result.

Command Line

The command line at the bottom accepts two kinds of input:

  • Shell commands — Recognized commands (ls, git, grep, etc.) are executed in your shell. Output is shown in the feedback area.
  • Natural language — Anything that doesn't look like a shell command is routed to the AI assistant (if configured).

As you type, Farx uses the AI to show typeahead suggestions in ghost text. Press Right to accept.

Configuration

Farx looks for config.toml in these locations (first match wins):

  • ~/.config/farx/config.toml
  • $XDG_CONFIG_HOME/farx/config.toml
  • ~/Library/Application Support/farx/config.toml (macOS)

All settings are optional — missing keys use defaults.

Full Reference

[general] confirm_delete = true # Prompt before deleting confirm_overwrite = true # Prompt before overwriting show_hidden_files = false # Show dotfiles use_trash = true # Move to trash instead of permanent delete editor = "internal" # "internal" or external command (e.g. "vim") viewer = "internal" # "internal" or external command (e.g. "less") [ui] theme = "tokyo-night" # far-classic, tokyo-night, catppuccin, dracula, gruvbox tick_rate_ms = 250 # Refresh interval in milliseconds show_fn_bar = true # Show function key bar at bottom date_format = "%Y-%m-%d %H:%M" # strftime format [panels] directories_first = true # Directories always sorted before files default_sort = "name" # name, extension, size, date [ai] enabled = false provider = "openrouter" # anthropic, openrouter, or openai-compatible base_url = "https://openrouter.ai/api/v1" model = "google/gemma-3-4b-it:free" max_tokens = 4096 api_key_env = "OPENROUTER_API_KEY" # Env var name containing the key

AI Setup

Farx supports multiple AI providers. Set one up in your config and export the API key as an environment variable.

OpenRouter (free tier)

[ai] enabled = true provider = "openrouter" base_url = "https://openrouter.ai/api/v1" model = "google/gemma-3-4b-it:free" api_key_env = "OPENROUTER_API_KEY"
export OPENROUTER_API_KEY="sk-or-..."

Anthropic (Claude)

[ai] enabled = true provider = "anthropic" base_url = "https://api.anthropic.com/v1" model = "claude-sonnet-4-20250514" api_key_env = "ANTHROPIC_API_KEY"

OpenAI

[ai] enabled = true provider = "openai-compatible" base_url = "https://api.openai.com/v1" model = "gpt-4o-mini" api_key_env = "OPENAI_API_KEY"

Ollama (local, private)

[ai] enabled = true provider = "openai-compatible" base_url = "http://localhost:11434/v1" model = "llama3.2" api_key_env = "OLLAMA_API_KEY"

For Ollama, set OLLAMA_API_KEY to any non-empty value (e.g. export OLLAMA_API_KEY=local).

Architecture

Farx uses a layered, action-based architecture. Every user interaction is mapped to an Action enum variant, which is dispatched by the central App struct to mutate state. The render loop then draws the new state.

┌──────────────────────────────────────────────────────────────┐
│  Terminal (crossterm)                                        │
│  ├─ Key / Mouse / Resize events                             │
│  └─ Periodic Tick events                                     │
└────────────────────────┬─────────────────────────────────────┘
                         ▼
┌──────────────────────────────────────────────────────────────┐
│  EventHandler (farx-ui/src/event.rs)                         │
│  Async bridge: crossterm → tokio channel                     │
└────────────────────────┬─────────────────────────────────────┘
                         ▼
┌──────────────────────────────────────────────────────────────┐
│  App::handle_key_event (farx-ui/src/app.rs)                  │
│  Routes input to deepest active modal, or to KeyMap          │
└────────────────────────┬─────────────────────────────────────┘
                         ▼
┌──────────────────────────────────────────────────────────────┐
│  KeyMap::resolve_panel (farx-core/src/keymap.rs)             │
│  Maps key combo → Action enum                                │
└────────────────────────┬─────────────────────────────────────┘
                         ▼
┌──────────────────────────────────────────────────────────────┐
│  App::dispatch(action) (farx-ui/src/app.rs)                  │
│  Mutates App state: tree, panels, modals, feedback           │
└────────────────────────┬─────────────────────────────────────┘
                         ▼
┌──────────────────────────────────────────────────────────────┐
│  App::render(frame) (farx-ui/src/app.rs)                     │
│  Draws TUI frame from current state                          │
└──────────────────────────────────────────────────────────────┘

Crate Structure

CratePurposeDepends on
farx-coreConfig, keymaps, actions, types, tree state(none)
farx-fsFilesystem operations (copy, move, delete, read)farx-core
farx-aiMulti-provider AI agentfarx-core
farx-uiTUI components, themes, event handlingfarx-core, farx-fs, farx-ai
farx-pluginPlugin system (WIP)farx-core
farx-appBinary, CLI parsing, auto-updatefarx-core, farx-fs, farx-ui

Rule: farx-core has zero internal dependencies. Lower crates never import higher ones.

Event Flow

When a key is pressed:

  1. crossterm emits a raw terminal event
  2. EventHandler wraps it as Event::Key(KeyEvent) and sends it through a tokio channel
  3. main.rs receives the event and calls app.handle_key_event(key)
  4. handle_key_event checks modals in priority order (editor > viewer > feedback > help > menu > search > AI bar > dialog > command line). The deepest active modal consumes the event.
  5. If no modal is active, KeyMap::resolve_panel(key) maps the key to an Action
  6. app.dispatch(action) mutates the application state
  7. On the next loop iteration, app.render(frame) draws the updated state

Contributing

See the full CONTRIBUTING.md on GitHub. The short version:

  1. Fork and clone the repo
  2. Create a feature branch
  3. Make your changes (the pre-commit hook runs cargo fmt and cargo check)
  4. Push and open a PR against main

Adding a Theme

Edit crates/farx-ui/src/theme.rs:

  1. Add a constructor function that returns Self with all fields set (copy an existing theme as a starting point)
  2. Register it in by_name() match statement
  3. Add it to available() array
pub fn my_theme() -> Self { let bg = Color::Rgb(30, 30, 40); let fg = Color::Rgb(200, 200, 210); Self { name: "my-theme", panel_bg: bg, panel_fg: fg, // ... all other fields ... } }

Adding a Keybinding

  1. Add the Action variant in crates/farx-core/src/action.rs
  2. Map a key to it in KeyMap::far_defaults() in crates/farx-core/src/keymap.rs
  3. Handle the action in App::dispatch() in crates/farx-ui/src/app.rs

Adding a UI Component

  1. Create the component file in crates/farx-ui/src/components/
  2. Export it in components/mod.rs
  3. Add state to the App struct
  4. Handle its key events in App::handle_key_event()
  5. Render it in App::render()

Follow the existing pattern of state struct + render function + action enum (see editor.rs or search.rs for examples).

Adding an AI Provider

Edit crates/farx-ai/src/agent.rs:

  1. Add a variant to ApiProvider enum
  2. Create request/response structs for the provider's API format
  3. Handle the new variant in query() and suggest() methods
  4. Map the provider name string in AiAgent::new()
